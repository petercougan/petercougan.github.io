Introduction

What are GitHub, Git, and Version Control?

  • GitHub is a software that runs on top of the system Git
  • Git is a system that defines a standardized way of handling version control
  • Version control is the idea that projects need a standardized method for handling changes. This is useful for:
    • Keeping past versions of files, in case there is a bug in the most recent
    • Multiple people working on the same file simultaneously
    • Tracking who has made what changes to what files
Basic Commands

What you need to know to start using GitHub.

clone

Copies and pastes the repository and its files from GitHub onto your local machine.
git clone https://github.com/{insert username}}/{insert repository name}.git

status

Tells you the difference between your local version of the repository and the latest version that is on the GitHub cloud. Also summarizes the set of changes to be commited.

add, add ., add -A

Maually selects an individual file to be added to the changeset.
. everything in directory, -A all
git add {insert file name here}

commit -m

Locks the change into the repository, but does not yet sync it with the GitHub cloud.
git commit -m '{insert description of commit here}'

push*
Syncs the committed changes on your local machine to the GitHub cloud.
pull*
Syncs the committed changes from the GitHub cloud to your local machine.

General Flow

Learn how to start using GitHub for version control from creating a new repository to syncing your changes with your team's changes.

  • Start From Scratch
    Getting started with a new repository and work environment

    1. Create a repository
      • Click 'New' under repositories on GitHub.com
      • Check the 'Initialize with a README' box
        • This will allow to immediately clone the repository
    2. Clone the repository
      • Retrieve the URL under the 'Clone or Download' dropdown on the main page for the repository
      • Open the Mac Terminal / Windows Command Prompt and change to the directory you wish to clone the project into
      • Enter git clone {insert URL here}

  • Main Flow
    Keeping your changes synced with your team member's changes

    1. Navigate to Local Directory
      • Using the Terminal/Command Prompt, go into the already existing directory for your repository
      • For example, if my repository was stored in ../../GitHub/Repositories/MyProject, I would change into the MyProject directory
    2. Grab All Changes Made By Team
      • This will help to prevent merge conflicts when you and another member of your team both have unchecked in changes for the same files
      • git pull
    3. Make Changes
      • Create, edit, or delete any files that you need to
      • As a general rule of thumb, you will want to only make changes for the specific task you are working on so you can make a commit (the next step) with a descriptive comment.
    4. Commit Changes to a Changeset
      • This will group all of your changes for the task you are working on, so you have a clean history of changes
        1. git add -A, git add {insert filename}, or git add .
        2. git commit -m "{insert description}"
    5. Repeat Steps 3 and 4
      • Make and commit changes frequently, as you complete each task
      • At this point, any changes between you and your team will still have not been synced
    6. Grab All Changes Made By Team, Again
      • At this point, if you and a team member have both edited the same lines of the same file, a merge conflict will show, or the pull will succeed
      • git pull
    7. Sync Your Changes
      • Assuming you have no merge conflicts, you will now need to get all of the changes you've made synced with the repository on the cloud
      • git push

  • Resolving Merge Conflicts
    Handling issues when you and a team member have edited the same lines of the same files. These steps should be followed between the last two steps of the Main Flow section

    1. Manually select which changes are the correct ones that should be pushed to the repository
      • GitHub won't know if your changes, your colleague's changes, or a combination of the two are the appropriate changes to be synced, so it will show you the two lines that are causing the conflict, and you must edit the file as needed
      • Consider the following line of code that is being edited between two people in their local repositories:
        <p>Hello, world!</p>
        You make the following edit:
        <p>Hello, universe!</p>
        Your colleague makes the following edit:
        <p class="welcome">Hello, world!</p>
        Then GitHub will provide the following code for you to merge:
        <<<<<<< HEAD
        <p>Hello, universe!</p>
        ========
        >>>>>>> ###############################
        <p class="welcome">Hello, world!</p>

        You can then merge the two like so, or simply keep one of the lines:
        <p class="welcome">Hello, universe!</p>
    2. Commit Changes to a Changeset
      1. git add -A
      2. git commit -m "{insert message here}"
    3. Sync Changes to Cloud
      1. git pull
      2. git push
